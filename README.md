# PRISMA Framework Pipeline README

## Overview

This repository contains the end-to-end training and evaluation pipeline used in the PRISMA framework. The workflow includes supervised fine-tuning (SFT), pseudo-label generation, DPO data construction, DPO training, reward fine-tuning data preparation, final SFT training, and evaluation on dev/test splits.

## What this README covers

Each row in the pipeline file has been converted into an execution step with the following information: task, script name, configuration file, command to run, expected inputs, and expected outputs.

## Requirements

- A working Conda environment named `trl_env`.

- Access to the project directory: `/DPO-ST-P/dpo-st-cop`.

- GPU-enabled training setup for `accelerate` and `torchrun`.

- Training and evaluation data in GSM8K JSONL format.

- Hydra configuration files under `exp_config/llama/` and accelerator config under `acc_config/fsdp.yaml`.

## Dataset Availability

A **sample dataset** is included in this repository to help you quickly understand and run the PRISMA pipeline.

For access to the **full dataset**, please use the following link:
[https://ai-nlp-ml.github.io/resources.html]

## Pipeline Summary

The pipeline is organized into three main phases:
1. **Base SFT**: fine-tune the pretrained LLaMA model on GSM8K.
2. **Preference data creation and DPO training**: sample multiple generations, evaluate them, build DPO/RFT datasets, and train the DPO model.
3. **Final SFT and evaluation**: train the final model with labeled + pseudo-labeled data and evaluate on dev/test splits.

## Step-by-Step Notes

### 1. Environment setup

Activate the Conda environment and move into the project directory before running any training or evaluation command.

```bash
conda activate trl_env
cd /DPO-ST-P/dpo-st-cop
```

### 2. Base model fine-tuning

The first training stage fine-tunes LLaMA on GSM8K using `sft.py` with the `sft-0` Hydra configuration and `accelerate`.
The resulting checkpoint folder is reported as `/DPO-ST-P/dpo-st-cop/ft_models/llama-2/sft-0`.

```bash
ACC_CONFIG='acc_config/fsdp.yaml'
accelerate launch --config_file $ACC_CONFIG sft.py --config-path=exp_config/llama --config-name=sft-0
```

### 3. Pseudo-label generation from the SFT model

For training preference data, the pipeline generates multiple sampled outputs per example. The pipeline sets `ARGS` to use the training split, sampling mode, and a maximum of 5 seeds.

```bash
ARGS='+data.split="train" eval.mode="sampling" eval.sampling.max_seed=5'
torchrun_path --nproc_per_node 8 generate_1.py --config-path=exp_config/llama --config-name=sft-0 $ARGS
```

Outputs are written under `model_outputs_nego_all/llama-2/sft-0/train/seed_{i}-t_0.7.json` for `i = 0..4`.

### 4. Preference-data evaluation and DPO dataset creation

`eval_sampling-1.py` compares the sampled generations against gold GSM8K answers and produces the initial DPO dataset.

```bash
python eval_sampling-1.py --config-name=sft-0 $ARGS
```

Then `utils/make_dpo_data-1.py` processes the raw DPO data into train/eval JSONL files.

```bash
python utils/make_dpo_data-1.py --config-path=/DPO-ST-P/dpo-st-copy/exp_config/llama --config-name=sft-0
```

### 5. DPO model training

```bash
ACC_CONFIG='acc_config/fsdp.yaml'
accelerate launch --config_file $ACC_CONFIG dpo.py --config-path=exp_config/llama --config-name=dpo-1
```

This stage consumes the processed DPO train/eval files and produces the DPO-trained LLaMA model.

### 6. Sampling pseudo-labels from the DPO model

The pipeline then samples a smaller set of pseudo-labels from the DPO model for additional training signal.

```bash
ARGS='+data.split="train" eval.mode="sampling" eval.sampling.max_seed=3'
torchrun_path --nproc_per_node 1 generate_1.py --config-path=exp_config/llama --config-name=dpo-1 $ARGS
```

### 7. Evaluation of DPO outputs against gold responses

```bash
python eval_sampling-1.py --config-path=exp_config/llama --config-name=dpo-1 $ARGS
```

### 8. RFT dataset generation

`utils/make_rft_data-1.py` transforms the DPO outputs into the reward fine-tuning (RFT) dataset format.

```bash
python utils/make_rft_data-1.py --config-path=/home/priyanshu/Prajwal/DPO-ST-P/dpo-st-copy/exp_config/llama --config-name=dpo-1
```

### 9. Final SFT training

The final supervised fine-tuning stage uses labeled and pseudo-labeled data from the RFT processing step.

```bash
ACC_CONFIG='acc_config/fsdp.yaml'
accelerate launch --config_file $ACC_CONFIG sft.py --config-path=exp_config/llama --config-name=sft-1
```

### 10. Evaluation on dev/test splits

Set the config path and split, generate greedy decoded pseudo-labels, and then run the final evaluation script.

```bash
CONFIG_PATH='exp_config/llama'
SPLIT='test'   # or 'dev'
which torchrun
# use the printed path in place of torchrun_path

torchrun_path --nproc_per_node 1 generate_1.py --config-path=$CONFIG_PATH --config-name=dpo-1 +data.split=$SPLIT
python eval_greedy-1.py --config-path=$CONFIG_PATH --config-name=dpo-1 +data.split=$SPLIT
```

## Output Locations

- Fine-tuned SFT checkpoint: `/DPO-ST-P/dpo-st-cop/ft_models/llama-2/sft-0`
- SFT sampling outputs: `model_outputs_nego_all/llama-2/sft-0/train/`
- Raw DPO data: `model_outputs_nego_all/llama-2/sft-0/train/train_dpo_data.jsonl`
- Processed DPO data: `model_outputs_nego_all/llama2/sft-0/train/train_dpo_processed.jsonl` and `model_outputs_nego_all/llama2/sft-0/train/eval_dpo_processed.jsonl`
- DPO sampling outputs: `model_outputs_nego_all/llama-2/dpo-1/train/`
- RFT processed data: `/model_outputs_nego_all/llama-2/dpo-1/train/train_rft_processed.jsonl`
- Final evaluation results: `results/llama-2.json`

## Important Notes

- The pipeline file contains both `llama-2` and `llama2` directory variants. Verify the exact directory name in your repository before running the scripts.
- Some commands use `torchrun_path` as a placeholder. Replace it with the actual path returned by `which torchrun` if needed.
- Several output paths start with a leading `/`, while others are relative. Check whether the repository expects absolute or relative paths in your environment.
- A few row descriptions appear duplicated or slightly inconsistent in the source file. The README preserves the intent of the pipeline while normalizing the wording for clarity.

## Suggested Execution Order

Run the steps in the same order as listed above: environment setup, base SFT, pseudo-label generation, DPO data processing, DPO training, DPO sampling, RFT dataset generation, final SFT, and final evaluation.
